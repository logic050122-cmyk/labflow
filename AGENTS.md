# AGENTS.md

## 1. 项目定位

本项目是实验室 / 工作室项目协作管理平台 LabFlow。

核心流程：

用户注册登录 → 创建项目或加入项目 → Owner 创建任务并分配成员 → Member 处理任务 → Member 提交完成 → Owner 审核 → 系统更新进度 → 发送通知 → 记录操作日志。

第一版目标是完成结构清晰、功能完整、便于答辩和后续扩展的单体 Web 应用。

当前优先级：

功能跑通 > 能看懂 > 基本分层 > 后续优化 > 高级工程化。

## 3. 当前功能范围

当前阶段只实现 LabFlow 的核心业务闭环：

用户注册登录 → 创建项目或加入项目 → 创建任务 → 分配任务 → 处理任务 → 提交完成 → 审核任务 → 更新状态 → 通知提醒 → 操作记录。

暂不主动实现以下功能：

1. 高级无障碍细节。
2. 深度权限配置。
3. 复杂统计分析。
4. 复杂筛选、排序、搜索。
5. 批量操作。
6. 高级消息通知。
7. 个性化设置。
8. 复杂通用组件封装。
9. 非核心页面美化和动效。
10. 非核心流程必需的额外功能。

如果某个功能不是核心流程必需内容，不要直接实现，可以放到后续建议中。

## 4. 角色规则

系统只保留项目角色：

* Owner：项目负责人
* Member：项目成员

规则：

1. 用户创建项目后，在该项目中是 Owner。
2. 用户加入项目后，在该项目中是 Member。
3. 权限判断必须基于当前项目角色。
4. 不做 Admin、Maintainer、Super Admin。
5. 前端隐藏按钮不等于权限控制，后端必须校验权限。

---

## 5. 任务状态规则

任务状态：

* todo：待开始
* doing：进行中
* submitted：待审核
* done：已完成
* overdue：已逾期

状态流转：

1. 创建任务后默认 todo。
2. Member 开始处理后变为 doing。
3. Member 提交完成后变为 submitted。
4. Owner 审核通过后变为 done。
5. Owner 审核驳回后回到 doing。
6. 超过截止时间且未完成，可由定时任务标记为 overdue。
7. 状态流转必须在 service 中校验。

---

## 6. 后端分层规则

后端按业务模块组织代码。

每个模块建议包含：

* xxx.routes.ts
* xxx.controller.ts
* xxx.service.ts
* xxx.repository.ts
* xxx.types.ts
* xxx.validator.ts

职责：

1. routes 只定义路由。
2. controller 只处理 req、res、参数获取和响应返回。
3. controller 不允许写复杂业务逻辑。
4. controller 不允许直接写 SQL。
5. service 负责业务规则、权限判断、状态流转。
6. repository 只负责数据库操作。
7. repository 不允许写业务判断和权限判断。
8. validator 负责参数校验。
9. types 负责类型定义。
10. middleware 负责认证、权限、错误处理。

禁止绕过 service 直接在 controller 中操作数据库。

---

## 7. 前端分层规则

前端按功能模块组织代码。

目录职责：

* views：页面级组件
* components：可复用组件
* api：接口请求封装
* stores：Pinia 状态管理
* router：路由
* types：类型定义
* utils：通用工具函数

规则：

1. 页面中不要直接写 axios。
2. 所有请求必须放在 api 目录。
3. 全局状态放在 stores。
4. 重复 UI 可以抽成 components。
5. 状态文本、状态颜色、状态枚举应统一维护。
6. 页面文件超过 300 行时，需要提示是否拆分。
7. 不要提前封装复杂通用组件、复杂 hooks 或复杂工具类。

---

## 8. 接口规则

1. 后端接口统一返回格式。
2. controller 不允许随意自定义返回结构。
3. 新增接口必须更新 docs/API.md。
4. 分页接口必须包含 list、total、page、pageSize。
5. 错误信息必须明确，不能只返回 error。
6. 接口字段应与 docs/API.md 保持一致。

详细接口清单、错误码、请求参数、响应字段，以 docs/API.md 为准。

---

## 9. 数据库规则

1. 表名使用小写复数形式。
2. 字段名使用 snake_case。
3. 主键统一使用 id。
4. 外键使用 xxx_id。
5. 核心表必须有 created_at 和 updated_at。
6. 修改数据库必须更新 docs/DATABASE.md。
7. 不允许代码中使用文档里没有的字段。

详细表结构、字段含义、索引、状态值，以 docs/DATABASE.md 为准。

---

## 10. 文档维护规则

开发过程中需要维护以下文档：

* docs/API.md：接口文档
* docs/DATABASE.md：数据库设计
* docs/DEV_LOG.md：开发日志
* docs/TODO.md：待办事项

规则：

1. 新增接口，必须更新 docs/API.md。
2. 修改接口字段，必须更新 docs/API.md。
3. 修改数据库表、字段、状态或索引，必须更新 docs/DATABASE.md。
4. 完成功能开发、业务缺陷修复、接口变更、数据库变更、依赖变更、配置变更或架构变更后，必须更新 docs/DEV_LOG.md。
5. 完成 TODO 后，必须更新 docs/TODO.md。
6. 仅修改注释、格式、文案、兼容性类型声明，且不改变运行行为和开发约定时，可以不更新 docs/DEV_LOG.md。

不要完整阅读 docs/DEV_LOG.md，除非需要查询历史问题。

## 11. 开发规则

1. 每次只开发一个模块。
2. 不允许一次性开发多个大模块。
3. 不允许随意重构全项目。
4. 不允许修改无关文件。
5. 修改前先阅读 AGENTS.md 和 docs 目录相关文档。
6. 修改前先说明开发计划。
7. 修改后必须说明修改了哪些文件。
8. 修改后必须说明如何测试。
9. 如果需求不明确，先提出问题，不要直接猜。
10. 如果代码可能变臃肿，先提示重构建议，不要直接大改。
11. 不允许用 mock 数据冒充真实接口完成。
12. 不允许随意引入新依赖。
13. 不允许未经说明修改认证逻辑。
14. 不允许未经说明修改数据库连接配置。
15. 不允许未经说明修改全局路由、全局状态、全局样式。
16. 不允许为了让编译通过而注释掉重要逻辑。
17. 不允许忽略 TypeScript 类型错误。

---

## 12. 代码风格

当前项目维护者处于初级程序员阶段。代码应优先保证直观、容易阅读、方便讲解。

要求：

1. 保留基本分层，但初期不要追求过度封装。
2. 优先使用简单明确的写法。
3. 不为了显得专业而使用复杂设计模式。
4. 不提前做复杂工程化。
5. 不提前封装复杂通用组件、复杂 hooks、复杂工具类。
6. 代码必须能运行、能理解、能讲清楚调用链。
7. 可以先写最小可用版本，后续再优化结构。
8. 不允许为了简单而破坏基本分层。
9. 不允许把 SQL、权限判断、业务逻辑全部混在一起。
10. 少用初学者难理解的高级语法；如果必须使用，需要解释清楚。

---

## 13. 代码臃肿提醒

出现以下情况时，先提醒用户，不要直接大改：

1. 单个文件超过 300 行。
2. controller 中出现大量 if/else 业务判断。
3. service 函数超过 80 行。
4. repository 中出现权限判断。
5. Vue 页面中直接写 axios。
6. 多个页面出现重复表单。
7. 多个地方重复写状态映射。
8. utils 中出现业务逻辑。
9. 一个模块依赖过多其他模块。
10. 一个功能需要修改超过 10 个文件。

## 16. 代码实现说明要求

每次完成功能开发后，必须用初学者能理解的方式简短说明具体代码实现逻辑。

说明内容包括：

1. 本次封装了什么函数、组件、接口或模块。
2. 为什么要封装。
3. 哪个文件调用哪个文件。
4. 用户点击按钮或触发操作后，代码执行顺序是什么。
5. 前端数据从页面到 api，再到后端接口的传递过程。
6. 后端数据从 routes、controller、service、repository 到数据库的执行过程。
7. 返回结果如何回到页面并展示。
8. 如果使用了稍难的语法，例如泛型、emit、Pinia、axios、async/await、try/catch，需要用简单语言解释用途。
9. 不要只说“调用接口”“封装请求”，必须说明具体调用链。
10. 说明要简短清楚，不要写成长篇文档。

说明格式：

```md
## 本次封装：

## 调用链：

## 数据流：

## 关键语法：

## 我需要重点看懂的文件：
```

## 18. Codex 工作原则

每次接到开发任务时，先判断任务属于哪个模块，再按当前模块最小范围修改。

优先做到：

1. 功能能跑通。
2. 文件职责清楚。
3. 调用链能讲明白。
4. 代码适合初学者阅读。
5. 文档同步更新。
6. 不扩大任务范围。
7. 不主动做复杂优化。

如果发现现有代码结构不合理，应先说明问题和简单替代方案，再等待用户确认是否重构。
